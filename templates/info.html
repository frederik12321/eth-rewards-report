How It Works

What you need, how to use it, and how your rewards are calculated

What You Need

To generate a report, you need two things:

  1. Your withdrawal address or validator index(es) — identifies your validator(s) on the Ethereum network. You can find your validator index on beaconcha.in by searching for your address.
  2. A free Etherscan API key — needed to fetch your on-chain transaction data. Get one at etherscan.io/apis (takes 30 seconds, free tier is sufficient).

Reporting Modes

The tool supports two reporting modes. You can switch between them using the toggle on the main page.

Default

Withdrawal-Based

Income is reported when ETH is swept to your address. Uses a cash-basis approach: each withdrawal event is a taxable moment, valued at the ETH price when it arrives.

Best for 0x01 (distributing) validators, where rewards are swept every ~5 days.

Advanced

Accrual-Based

Income is reported as it accrues daily on the beacon chain, even before it's withdrawn. Tracks the daily balance change for each validator.

Recommended for 0x02 (compounding) validators, where rewards stay in the validator balance. Also useful in jurisdictions that require accrual-based tax reporting.

When to use accrual mode

  • Your validators use 0x02 compounding credentials (rewards compound instead of being swept)
  • Your jurisdiction treats staking rewards as income when earned, not when received
  • You want to track daily reward accumulation independently of withdrawal timing
Accrual mode requires a beaconcha.in API key to fetch daily validator stats. Get a free key at beaconcha.in/api. The first run may take a few minutes for many validators (data is cached for subsequent runs).

Withdrawal Address vs Fee Recipient

Ethereum validators have two addresses that can receive rewards:

  • Withdrawal address — set in your validator's withdrawal credentials. Receives consensus layer (CL) reward sweeps automatically every ~5 days.
  • Fee recipient address — set in your validator client configuration. Receives execution layer (EL) rewards: priority fees and MEV payments when your validator proposes a block.

For most solo stakers, these are the same address. In that case, you only need to enter the withdrawal address and can leave the fee recipient field empty.

If you use a different address as your fee recipient (e.g. a hot wallet), enter it in the optional fee recipient field so the tool can find those rewards too.

Step by Step

1
Resolve validators
Your validator index, public key, or withdrawal address is resolved via the Beacon Chain API to identify which validators to track, their withdrawal address, and credential type (0x01 or 0x02).
2
Fetch historical prices
Hourly ETH prices in your chosen currency are fetched from CryptoCompare (with DeFiLlama and CoinGecko as automatic fallbacks). Prices are cached locally — subsequent reports reuse cached data.
3
Fetch reward events
Withdrawal mode: CL withdrawals are fetched from Etherscan for your withdrawal address. For 0x02 validators, withdrawals are automatically classified as income vs principal.
Accrual mode: Daily balance changes are fetched from beaconcha.in for each validator index. The daily reward is calculated as (end_balance - start_balance) - deposits + withdrawals. Events are aggregated by date across all validators.
4
Fetch execution layer rewards
Block proposals and MEV payments are fetched from Etherscan for your fee recipient address. MEV is detected via a list of ~40+ known builder addresses, with a fallback check for unknown senders. If Etherscan is rate-limited, Routescan and Blockscout are tried automatically.
5
Price matching
Each reward event is matched to the closest hourly ETH price, giving a fiat value accurate to within ~30 minutes.
6
Report generation
All events are compiled into a report with monthly breakdowns, totals, and per-validator statistics. Available as Excel (.xlsx), CSV, or both. The Excel report includes a summary sheet, monthly detail sheets, and a validators sheet.

Validator Types

0x01 Distributing Validators

The original withdrawal credential type. Max effective balance is 32 ETH. Any balance above 32 ETH is automatically swept to your withdrawal address roughly every 5 days. All withdrawals are income.

0x02 Compounding Validators

Introduced with the Pectra upgrade (May 2025). Rewards compound in the validator balance up to 2048 ETH effective balance. The tool automatically detects 0x02 validators and classifies withdrawals:

  • Skimming (income) — automatic sweeps of balance exceeding the 2048 ETH cap. Taxable income, similar to 0x01 validator sweeps.
  • User-requested (not income) — partial withdrawals you initiated via the EIP-7002 system contract. Treated as return of capital.
  • Consolidation transfers (not income) — balance moves between validators via EIP-7251. Internal transfer, not new income.
Recommendation: For 0x02 compounding validators, consider using accrual-based reporting to capture daily reward accumulation, since rewards stay in the validator balance rather than being swept out.

Execution Layer Rewards

When your validator proposes a block, it earns execution layer rewards:

Local Block Rewards (Priority Fees)

When your validator builds its own block, the transaction priority fees are paid directly to your fee recipient address.

MEV Payments

Most validators use MEV-boost to outsource block building to specialized builders for higher rewards. The builder sends a lump-sum payment to your fee recipient address. The tool detects these by:

  1. Checking the sender against a list of ~40+ known builder addresses (instant)
  2. For unknown senders: looking up the block's fee recipient to verify the payment came from whoever built the block (requires an extra API call per candidate)

Price Data

Historical ETH prices are sourced at hourly resolution. The tool tries multiple sources in order:

  1. CryptoCompare — primary source, supports 10+ fiat currencies directly
  2. DeFiLlama — automatic fallback, uses ETH/USD with ECB forex conversion via Frankfurter API
  3. CoinGecko — second fallback, useful for currencies not covered by CryptoCompare

Prices are cached locally after the first fetch, so subsequent reports for the same period load instantly.

Supported currencies

EUR, USD, GBP, CHF, CAD, AUD, JPY, SEK, NOK, DKK — and more via CoinGecko fallback.

Note: Prices represent a volume-weighted average across major exchanges. Small differences compared to other tools are normal due to different price sources and time resolution.

Output Formats

Excel (.xlsx)

The Excel report includes multiple sheets:

  • Summary — monthly totals with separate columns for accrual rewards, withdrawals, block rewards, and total income (in both ETH and your fiat currency)
  • Validators — per-validator details including credential type, status, balance, and total withdrawn
  • Monthly sheets — one sheet per month with every individual reward event, its timestamp, ETH amount, price, and fiat value

CSV

A flat file with one row per reward event. Includes date, time, type, validator, ETH amount, price, and fiat value. Easy to import into accounting software or spreadsheets.

Both

Downloads a ZIP file containing both the Excel and CSV reports.

Timezone: You can set a timezone in Advanced Options. All dates and times in the report will use your selected timezone instead of UTC.

Rate limits & reliability

  • Etherscan: 5 calls/sec on free tier. If rate-limited, the tool automatically falls back to Routescan and Blockscout for execution layer queries.
  • beaconcha.in: 10 requests/min. Each validator requires one API call on first run. Data is cached — repeat runs are instant.
  • CryptoCompare: Generous free tier. Falls back to DeFiLlama/CoinGecko automatically if unavailable.
Caching: Both price data and validator stats are cached locally. After the first run, subsequent reports generate much faster since only new data needs to be fetched.

Privacy & Security

  • No data stored — your API keys, addresses, and report data are processed in memory and never saved to disk (except temporary report files that are deleted after download)
  • API keys masked — API keys are never included in log messages or error outputs
  • Server-side processing — all API calls are made from the server, so your API keys are never exposed in browser network traffic
  • HTTPS enforced — all connections use HTTPS with HSTS headers
